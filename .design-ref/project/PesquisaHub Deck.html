O problema

A gestão dos projetos de pesquisa
está espalhada em planilhas, e-mails e PDFs.

Coordenadores não têm visão única dos seus projetos. Bolsistas registram horas em formulários paralelos. A PROPESP só vê tudo no fim, quando precisa relatar para Embrapii, CNPq, FAPESPA e CAPES.

Sem fonte única da verdade Indicadores institucionais manuais Prestações de conta sob pressão

Um sistema, três experiências.

Cada perfil enxerga só o que precisa — com permissões e dados ajustados ao seu papel.

Protótipo navegável

Cadastro com campos dinâmicos
por tipo de projeto.

Um único formulário em 5 passos atende três realidades distintas. Para P&D Embrapii, aparecem TRL e contrapartida. Para Extensão, público-alvo e municípios. Para IC, modalidade da bolsa e plano de trabalho.

1 formulário · 3 tipos Validação por etapa Salvar como rascunho
Justificativa técnica

Por que MongoDB?

Três pilares justificam a escolha do banco para este domínio.

01

Modelagem
híbrida

Cada tipo de projeto — P&D, IC, Extensão — tem campos próprios. O schema flexível do Mongo absorve essas variações sem migrations a cada novo edital.

schema-on-read · sem ALTER TABLE
02

Hierarquia
de dados

Projeto → Metas → Entregas → Horas é uma árvore natural. Documentos aninhados servem essa estrutura em uma única consulta — sem JOINs em cascata.

embedded docs · 1 query, todo o contexto
03

Performance
com Node.js

Driver oficial nativo, JSON no banco e no front. Menos serialização, menos ORM, menos camadas — Express + Mongoose entregam respostas em milissegundos.

JSON end-to-end · driver oficial maduro
